

React?
복잡한 사용자 인터페이스를 쉽게 구축하기 위해 메타에서 개발한 라이브러리


React의 핵심 컨셉(가치)

    1.Component (컴포넌트)

    레고
    //html을 반환하는 함수
    한가지 이상의 기능을 수행하는 UI를 모듈화 
    각각의 페이지 구성에 필요한 component를 조힙하여 페이지를 완성한다

    => 이점 
    길어질 수 있는 코드에 대한 (기능별로 나누었기때문에) 가독성 해결  (단일책임원칙)
    반복되는 UI의 재사용
    단일 책임 원칙 (한가지의 기능) - 유지보수

    2. Virtual Dom (가상돔)

    DOM API를 사용하지 않습니다. 
    React가 가지고 있는 가상의 DOM Tree, 실제 사용자가 보고 있는 DOM 간섭X
    state(상태 값)을 구독, 상태 값이 바뀌었다면, 바뀐 부분만 화면을 다시 그림

    이점 

    reflow repaint 과정이 state가 변했을 떄 최초 1회만 시행
    상태값만을 구독하고 있기 때문에 화면을 바꾸는 시점을 개발자가 컨트롤 하기 쉬움
    dom api에 접근하지 못하는게 아니다 (지양) -> DOM API(X)

    3. ReRendering

    (컴포넌트가 함수기반)
       상태가 바뀌면 상태가 바뀐 함수만 재실행하기 떄문에 다른 함수(ui)는 
       다시 그려지지 않는다

    SPA, MPA 

    SPA(Single Page application) ----------(react)
    html 파일이 하나(도화지) 주소(URL)에 따라 필요한 컴포넌트를 도화지에 '조립'하는것

    MPA(Multi Page Application)
    html이 여러개

    CSR, SSR 

        CSR (Client Side Rendering)
            사용자가 웹 사이트에 접속했을 떄 페이지 렌더링에 필요한
            리소스를 다운로드하고 브라우저 엔진이 화면을 렌더링

        장점
           첫 렌더링 이후 전환 속도가 빠름
           서버 부담 감소로 인한 서버 비용 감소

        단점 
            초기 페이지 렌더링 속도가 느림
            검색 엔진 최적화(SEO)가 불리하다.
            사용자 하드웨어에 의존하여 화면을 그린다

        SSR (Server Side Rendering)
            사용자가 웹 페이지 접속 -> 프론트 엔드 서버에 요청
            -> 프론트 엔드 페이지가 그려질 수 있는 준비 

            장점 
                완성된 페이지를 보게되는 시간이 짧습니다.
                ------------(백엔드에 요청한 데이터)

                SEO(검색 엔진 최적화)
                <meta title=""/>

            단점 
                만약 백엔드에서 그리는 시간이 길어진다면 비어있거나 로딩화면을
                오래 볼 수 있다.(무거운 페이지라면 csr보다 오래 걸릴수있다)

                서버 부담 증가(서버의 하드웨어가 중요)

    패키지 관리 도구
        패키지(프로젝트를 실행하고있는 폴더)에서 실행할 수 있는 명령어를 관리하는 도구

        - 라이브러리 설치 및 삭제 
    
    
    npm,npx,yarn,berry,pnpm
    ---     ---->>기본 패키지 관리도구
            --------------->> 모노레포 관리 도구

    npm
        nodejs에서 내장되어있는 기본 관리 도구
        ex) react부터는 라이브러리를 사용할 때 CDN으로 사용하지 않습니다 --> 라이브러리를 설치
            패키지관리 도구를 사용하기 위해서는 경를 패캐지에 맞추는것이 중요하다.cd example
            패키지 -> 프로젝트를 진행중인 폴더

    
        npm i(install) 라이브러리명 (aos)
        npm rm(remove, uninstall) 라이브러리명
    
    npx (react 프로젝트를 만들때 한번 만!)
        nodejs에서 내장되어있는 도구 라이브러리를 설치하지 않아도 명령어를 통해 실행
        (특수한 상황이나 일회성 사용하는 라이브러리를 활용하거나 스크립트를 실행할 떄 사용)

    yarn
        실무에서는 가장 많이 사용하고 있는 도구, npm보다 보안이나 속도가 빠르다라는 장점

        npm i -g(global) yarn (컴퓨터에 설치)
        
        yarn add 라이브러리명
        yarn remove 라이브러리명
        
    하나의 프로젝트에서는 하나의 패키지 관리 도구만 사용

    리액트 폴더 구조
    > node_modules : 현재 패키지에 설치된 라이브러리의 실제 코드들이 저장 (다른 사람의 코드가 들어가있는곳)
                   : 용량이 큼 git에 올리지 않습니다 (제외)(프로젝트가 끝나면 지워도 됨)
                   : package.json이 있다면 언제든 재설치 --> npm install, yarn
                     node_modules가 삭제된 상태로 파일을 받았다면 node_modules를 재설치해야하기때문에 npm install을 해줘야한다
    
    > public            : 정적 파일을 관리, 로고, 페비콘, index.html, sitemap, robots (검색엔진)
    > src(source)       : 개발이 진행되는 공간 
    .gitignore          : git에 올리지 않을 파일
    package-lock.json   : 패키지에 설치된 라이브러리에 실제 설치 주소와 버전이 맵핑
    * package.json      : 실행 스크립트를 정의, 설치된 라이브러리 목록, 프로젝트 정보, 버전관리
    READEME.md          : mark down
